![机制图展示 AI 生成代码后依次进入安全扫描、证据工作区、修复建议、PR 验证和人工复核。](/img/posts/2026-07-01-topics/diagram.png)

*图：AI 写完代码后，安全证据需要进入扫描、findings、PR 验证和人工复核链路。*

AI 写代码已经不稀奇了。真正影响团队是否敢用的，是它写完以后有没有留下安全证据：发现了什么、为什么可疑、怎么修、谁复核、能不能进入 PR 和审计流程。

这也是最近值得注意的变化：AI 编程工具正在从“生成代码”走向“生成后的安全审查”。

OpenAI 的 Codex Security plugin 在 2026 年 6 月更新到 0.1.9，changelog 里明确提到几类能力：findings workspace、低设置成本扫描、JSON/CSV/SARIF 导出、既有 findings 的 triage，以及 issue/advisory 跟踪。[S01]

这些词看上去不如“模型更聪明”那么热闹，但对工程团队更关键。

因为团队真正怕的不是 AI 不会写代码，而是它写得很快、看起来能跑、测试也绿了，最后把一个业务逻辑漏洞、依赖风险或 secret 泄漏带进主干。

Codex Security quickstart 里给出的定位也很清楚：它不是在说“AI 代码已经安全”，而是把插件放在授权代码上运行，扫描 vulnerabilities，验证 plausible findings，并在 workspace 里呈现证据和修复建议。[S02]

这里的关键词不是“自动修好一切”，而是“证据”和“workspace”。

过去很多 AI coding 的交互是散的：一段聊天、一次 diff、一个建议、一个开发者的肉眼判断。问题是，安全审查最怕散。真正进入团队流程后，你需要知道：

- 这次扫描覆盖了什么代码；
- 哪些发现被认为 plausible；
- 每个 finding 的证据是什么；
- 修复建议是否能被复查；
- 结果能不能导出给 CI、审计或安全平台；
- 最后有没有人承担合并责任。

所以 JSON、CSV、SARIF 这类导出格式并不只是“多一个功能”。它们意味着 AI 生成后的风险可以离开聊天窗口，进入更严肃的工程系统。[S01]

GitHub 这边也出现了同一方向的信号。GitHub 在 2026-06-09 宣布，对第三方 coding agents 生成的代码默认进行安全验证，覆盖 CodeQL、GitHub Advisory Database、secret scanning 相关检查；如果发现问题，agent 会尝试在最终 PR 前修复。[S07]

这件事的含义也不是“第三方 Agent 写的代码从此安全了”。

更准确地说，它说明平台正在把 coding agent 当成正式开发参与者处理：既然它能提交代码，就要进入和人类开发者类似的验证链路。生成不是终点，进入 PR 前的检查才是工程化的开始。

可以把这条链路理解成这样：

AI 生成代码，安全扫描发现可疑点，findings workspace 保存证据和建议，开发者或 Agent 根据建议修复，再进入 PR 验证，最后由人做 review 和合并判断。

这条链路里，每一步都不能被吹成万能。

扫描器可能漏掉业务逻辑漏洞。secret scanning 也不能理解你的权限设计是否合理。CodeQL 能覆盖一批代码模式，但不能替代团队对产品场景的威胁建模。AI 给出的修复建议也需要复核，否则可能把一个问题改成另一个问题。

这正是这类工具最容易被误解的地方。

支持者会说：安全审查越早进入 AI 编码流程越好。开发者刚让 Agent 改完代码时，修改上下文还在，修复成本最低。如果等到上线前或事故后才查，风险已经被放大。

反对者也有道理：厂商插件和平台默认验证容易给人一种“扫过就安全”的错觉。尤其是团队刚开始使用 AI coding 时，速度提升带来的兴奋感，会掩盖权限、供应链、业务逻辑和数据边界这些更难自动化的问题。

所以我更愿意把这轮变化理解为：AI 编程进入工程化阶段的一块拼图。

以前大家问的是：“这个 Agent 能不能帮我写？”

现在更应该问：“它写完以后，能不能把风险带回我的流程？”

对一线开发者，最实用的判断方法不是看宣传页，而是看六个问题：

第一，它是否只在授权代码和明确范围内运行。[S02]

第二，它是否能把发现的问题整理成可复查的 findings，而不是只在聊天里说一句“这里可能有风险”。[S01][S02]

第三，它是否能输出机器可读结果，例如 SARIF，让结果进入 CI、安全平台或长期追踪。[S01]

第四，它是否能连接 issue、advisory 或 PR 流程，而不是停在一次性建议。[S01][S07]

第五，它是否明确告诉你自己的边界，比如不能证明所有漏洞、不能替代人工 review。

第六，团队是否规定了最终合并责任。AI 可以建议，扫描器可以提示，但上线责任不能丢给工具。

如果这六点做不到，AI coding 仍然可以用，但最好把它当成“辅助写代码”，而不是“可信的工程成员”。

如果这六点逐步补齐，AI coding 的价值就不只是快。它会开始变成一种更可审计的开发方式：每次生成、每次风险发现、每次修复建议、每次 PR 验证，都能留下痕迹。

这才是我觉得 Codex Security 和 GitHub 第三方 coding agent 验证值得写的原因。

它们不是在宣布 AI 写代码已经安全，而是在提醒我们：AI 写代码越快，安全证据越不能慢。

给工程团队的几个建议：

1. 先把 AI coding 的使用范围写清楚：哪些仓库、哪些目录、哪些数据不能碰。
2. 不要只保存最终 diff，也要保存 findings、扫描结果和修复理由。
3. 能导出 SARIF/JSON/CSV 的结果，尽量接进现有 CI 或安全平台。
4. 把第三方 coding agent 的 PR 当成正式代码变更处理，保留人工 review。
5. 对业务逻辑、权限边界、数据访问这类问题，不要期待通用扫描器自动兜底。

最后可以留一个问题：如果你所在团队已经开始让 AI 写代码，你们现在更缺的是“生成能力”，还是“生成后的安全证据链”？
